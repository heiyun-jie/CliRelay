# CliRelay 核心原则

## 1. 先确认链路，再改功能

凡是涉及 `/v1` 请求注入、中间件顺序、管理路由、面板代理、SQLite 写入的问题，先确认真实链路，再动实现。

## 2. 记忆是增强层，不是主业务逻辑

本地 memory、SCE 命中、Intent Upgrade 都只能作为增强上下文，不能破坏原始请求的可达性和稳定性。

## 3. 本地事实优先于历史印象

看到文档和代码冲突时，以当前仓库代码、当前配置、当前运行数据为准。

## 4. 接管时优先收口，不优先扩展

当前项目最需要的是稳定性和治理边界，不是继续无约束扩能力。

## 5. 先做最小闭环

每次推进优先形成一个可验证闭环：

1. 需求入口明确
2. 代码路径明确
3. 测试或验证方式明确
4. 文档同步更新

## 6. 记忆分层必须清楚

当前默认分层：

1. `conversation_turns`：短期对话连续性
2. `memory_entries`：本地长期记忆
3. `internal/sce`：项目知识检索

不要把这三层概念混成一个词。
