# 30-04 请求来源审计

## 1. 概述

当前项目已有“请求日志”页面，但其数据本质上是模型调用日志，并不包含稳定的来源机器审计字段，也没有独立的管理访问审计通道。

本 Spec 的目标是：

1. 让模型调用日志具备来源机器识别能力
2. 为 `/v0/management` 建立独立的访问审计
3. 在控制台中把“模型请求日志”和“管理访问审计”分开展示

## 2. 需求

### 2.1 模型请求日志补充来源字段

**验收标准**

1. `request_logs` 至少保存 `client_ip`、`forwarded_for`、`user_agent`
2. 新请求进入 SQLite 后，可以从日志中区分不同来源机器
3. 现有请求日志页面可展示来源字段，不破坏现有 token / 模型 / 渠道统计

### 2.2 管理访问审计独立落库

**验收标准**

1. 新增独立的管理访问审计表，不与 `request_logs` 混用
2. `/v0/management/*` 的成功与拒绝访问都可记录
3. 审计日志至少包含时间、IP、X-Forwarded-For、User-Agent、方法、路径、状态码、允许/拒绝结果、认证结果

### 2.3 控制台独立展示

**验收标准**

1. 控制台新增“访问审计”页面
2. “请求日志”页继续用于模型调用日志
3. “访问审计”页可查看最近管理访问来源

### 2.4 当前运行环境可观测

**验收标准**

1. 当前本地运行配置开启 `usage-statistics-enabled`
2. 当前本地运行配置开启 `request-log`
3. 修改后重启服务，新的请求可进入最新日志链路
